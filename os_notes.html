1.1 Computer System Overview
Computer System Components

A computer system has four major components working together:

  • Hardware — CPU, RAM, I/O devices — provides basic computing resources
  • Operating System — controls and coordinates hardware use among applications
  • Application Programs — word processors, compilers, web browsers, games
  • Users — people, machines, other computers
OS RoleThe OS is a resource allocator — it manages CPU time, memory space, file storage, I/O devices. It is also a control program — it controls execution of user programs to prevent errors and improper use.
CPU (Central Processing Unit)

The brain of the computer. Fetches instructions from memory, decodes them, executes them (fetch-decode-execute cycle). Contains:

  • ALU (Arithmetic Logic Unit) — performs arithmetic (+, -, ×, ÷) and logical (AND, OR, NOT) operations
  • Control Unit — directs operations of the processor, tells memory/ALU/I/O how to respond to instructions
  • Registers — small, fast storage inside CPU: PC (Program Counter), IR (Instruction Register), MAR, MBR, SP (Stack Pointer), PSW
  • Cache — L1, L2, L3 cache between CPU and RAM for speed
Main Memory (RAM)

Random Access Memory — volatile storage that directly interacts with CPU. Programs must be loaded into RAM before execution. Organized as an array of bytes, each with a unique address.

CPU can only process data that is in registers or RAM. Data on disk must be loaded into RAM first.
I/O Devices

Each I/O device is paired with a device controller (hardware) and a device driver (software in OS). Controller has local buffer registers and communicates with CPU via bus.

Interrupt Mechanism
InterruptA signal sent to the CPU by hardware or software indicating an event that needs immediate attention. Causes CPU to stop current task, save state, and run an Interrupt Service Routine (ISR).

How interrupts work (step-by-step):

  1. Hardware device raises an interrupt signal on the bus
  2. CPU finishes current instruction
  3. CPU saves current state (PC, registers) onto the stack
  4. CPU looks up Interrupt Vector Table (IVT) to find ISR address
  5. CPU transfers control to ISR
  6. ISR executes (handles the event — e.g., reads key pressed)
  7. CPU restores saved state and resumes previous task
📌 Example: Keyboard Interrupt

You press a key → keyboard controller puts the key code in its buffer → sends interrupt to CPU → CPU pauses current process → runs keyboard ISR → ISR reads key from buffer → stores in OS buffer → returns to interrupted process

Dual-Mode Operation
Dual-Mode OperationHardware provides at least two modes of operation — User Mode and Kernel Mode — to protect OS from errant user programs.

A mode bit in the CPU (0 = Kernel/Privileged, 1 = User) indicates current mode.

FeatureUser ModeKernel Mode
Mode bit10
Privileged instructions❌ Not allowed✅ Allowed
Direct hardware access❌✅
Who runs hereUser applicationsOS kernel code
Memory accessLimited (own address space)Full memory

Transition from User → Kernel mode: only via system call, interrupt, or exception. Transition from Kernel → User mode: OS sets mode bit to 1 before returning to user program.

Timer

A timer is used to prevent a user program from running forever (infinite loop). The OS sets a timer to interrupt after a fixed time period. If the timer fires, control returns to OS. This enables time-sharing.

📌 Example

OS sets timer to 10ms. Process P1 starts. After 10ms timer fires → interrupt → OS regains control → switches to P2. This is how Round Robin scheduling works at hardware level.

A trap is generated when:

B. A process executes a division by zero

Q3 — GATE Style | 2 Marks
In a system with dual-mode operation, which of the following instructions should be privileged (kernel mode only)?
I. Read from a file   II. Set the timer interrupt   III. Clear the interrupt flag   IV. Issue a system call
A. I and III only
B. II and III only
C. I, II and III

Privileged instructions are those that can disrupt system operation if misused. Setting the timer (II) and clearing interrupt flags (III) are privileged — a user program doing these could disable OS preemption. Issuing a system call (IV) is a user-mode instruction (it triggers the trap); the actual work happens in kernel mode.
1.2 OS Structures & Architecture
System Call Interface
System CallA programmatic way for a user process to request a service from the OS kernel. It provides the interface between a running program and the OS.

System call execution steps:

  1. User program calls library function (e.g., read() in C)
  2. Library function puts system call number in register and executes TRAP/INT instruction
  3. CPU switches to kernel mode
  4. OS looks up system call table using the number
  5. OS executes the system call handler
  6. OS returns result, CPU switches back to user mode
  7. Library function returns result to user program

Process States

5-State Model:

  • New — Process being created (PCB created, not yet admitted)
  • Ready — Waiting in ready queue; has all resources except CPU
  • Running — Currently executing on CPU (only 1 at a time on single-core)
  • Waiting/Blocked — Waiting for I/O or event (not using CPU, not ready)
  • Terminated — Finished execution; PCB cleaned up

State transitions:

  • New → Ready: Process admitted by long-term scheduler
  • Ready → Running: CPU dispatched by short-term scheduler
  • Running → Ready: Preempted (timer interrupt) or yield
  • Running → Waiting: I/O request or event wait
  • Waiting → Ready: I/O completed or event occurred
  • Running → Terminated: Process exits normally or killed
GATE frequently asks about which transitions are possible. Running → Waiting is caused by I/O request (voluntary). Running → Ready is caused by preemption (involuntary). Ready → Running ONLY via scheduler — a process CANNOT go directly from Ready to Waiting.
Process Control Block (PCB)
PCB (Process Control Block)A data structure maintained by the OS for every process. It is the OS's representation of a process — the OS knows everything about a process through its PCB.

PCB fields (memorize these — exams love this!):

  • Process ID (PID) — unique identifier
  • Process State — new, ready, running, waiting, terminated
  • Program Counter (PC) — address of next instruction to execute
  • CPU Registers — all register values (saved during context switch)
  • CPU Scheduling Information — priority, scheduling queue pointers
  • Memory Management Information — page tables, segment tables, base/limit registers
  • I/O Status Information — list of I/O devices allocated, open files
  • Accounting Information — CPU time used, real time used, time limits

Context Switch
Context SwitchThe process of saving the state of the currently running process (into its PCB) and loading the state of the next process (from its PCB). During a context switch, the CPU does no useful work — it is pure overhead.
Context switch time is PURE OVERHEAD. Questions may ask to calculate effective CPU utilization accounting for context switch time.
Effective CPU Utilization: CPU useful time = Total time − (Number of context switches × Context switch time) Utilization = CPU useful time / Total time × 100%
CPU Scheduling Algorithms — All Types

Performance Metrics:

  • Arrival Time (AT) — when process enters ready queue
  • Burst Time (BT) — CPU time needed
  • Completion Time (CT) — when process finishes
  • Turnaround Time (TAT) = CT − AT
  • Waiting Time (WT) = TAT − BT
  • Response Time = first CPU allocation time − AT

3.4 Concurrency Issues
Race Condition
Race ConditionWhen multiple processes/threads access and manipulate shared data concurrently, and the outcome depends on the particular order of execution (scheduling order). Result is non-deterministic and can be incorrect.
📌 Classic Race Condition Example

Shared variable: counter = 5. Two threads both execute counter++.

Thread 1: load counter (5) → increment to 6 [preempted!]

Thread 2: load counter (5) → increment to 6 → store 6

Thread 1: store 6 (its old value)

Result: counter = 6 instead of 7! Lost update problem.

At machine code level, counter++ is 3 instructions: LOAD, ADD, STORE. Race occurs because this is not atomic.

4.1 Critical Section Problem
Critical Section
Critical SectionA segment of code where a process accesses shared resources (shared variables, files, tables). Only ONE process should execute in its critical section at a time.

Structure of a process with critical section:

do { [entry section] ← acquire permission to enter CS [critical section] ← access shared resource [exit section] ← release permission [remainder section] ← non-critical code } while(true);

3 Requirements for a valid solution:

  • 1. Mutual Exclusion — If process Pi is in CS, no other process can be in CS. (Safety requirement)
  • 2. Progress — If no process is in CS and some want to enter, only those NOT in remainder section can decide who enters next; this decision can't be postponed indefinitely. (No deadlock)
  • 3. Bounded Waiting — After process Pi requests entry to CS, there is a bound on how many times OTHER processes can enter before Pi gets to enter. (No starvation)
Peterson's Algorithm

Software solution for 2-process critical section. Uses two shared variables: int turn (whose turn it is) and boolean flag[2] (whether process wants to enter).

/* Process Pi (i=0 or 1, j = 1-i) */ do { flag[i] = true; // I want to enter turn = j; // but I'll let you go first while (flag[j] && turn == j); // wait if both want AND it's j's turn [critical section] flag[i] = false; // I'm done [remainder section] } while(true);

Why it works:

  • Mutual exclusion: Both in CS requires flag[0]=flag[1]=true AND turn=0 AND turn=1 simultaneously — impossible
  • Progress: If Pj doesn't want CS (flag[j]=false), Pi enters immediately
  • Bounded waiting: After Pi sets turn=j, if Pj wants to enter, Pj gets it ONCE, then turn=i, so Pi gets in next time
Peterson's doesn't work on modern CPUs with reordering/caching without memory barriers. GATE assumes sequential consistency.
Hardware Support: Test-and-Set
/* TestAndSet — atomic hardware instruction */ boolean TestAndSet(boolean *target) { boolean rv = *target; *target = true; // atomically set to true return rv; // return old value } /* Using TestAndSet for mutex */ boolean lock = false; do { while (TestAndSet(&lock)); // busy wait until lock was false [critical section] lock = false; [remainder] } while(true);
TestAndSet and Compare-and-Swap are atomic hardware instructions. They cannot be interrupted mid-execution. They solve mutual exclusion but do NOT guarantee bounded waiting by themselves.
4.4 Synchronization Tools
Semaphore
SemaphoreAn integer variable accessed only through two atomic operations: wait() (P operation / down) and signal() (V operation / up). Used for both mutual exclusion and process synchronization.
wait(S) { signal(S) { while (S <= 0); S++; S--; } } /* Both operations are ATOMIC (indivisible) */
TypeInitial ValueUse
Binary Semaphore (Mutex)1Mutual exclusion — only 1 process in CS at a time
Counting SemaphoreN (resource count)Resource management — up to N processes can access simultaneously
📌 Using Semaphore for Mutex
Semaphore mutex = 1; // binary Process P1: Process P2: wait(mutex); wait(mutex); [critical section] [critical section] signal(mutex); signal(mutex);
📌 Using Semaphore for Synchronization (ordering)

Ensure statement S2 in P2 executes AFTER S1 in P1:

Semaphore synch = 0; P1: P2: S1; wait(synch); signal(synch); S2;
Monitor
MonitorA high-level synchronization construct. A monitor is a class/module with shared variables, procedures, and an implicit mutex — only ONE process can be active inside the monitor at a time. Uses condition variables for waiting.
monitor MonitorName { // shared variables condition x, y; procedure P() { // mutual exclusion automatic if (not_ready) x.wait(); // suspend, release monitor // ... x.signal(); // wake one waiting process } }
x.wait() releases the monitor lock and suspends the process. x.signal() wakes up one process waiting on x. If no one waiting, signal() has no effect (unlike semaphore where it increments).
4.5 Classical Synchronization Problems
1. Producer-Consumer (Bounded Buffer)

Producer produces items into a bounded buffer. Consumer consumes from it. Need to: prevent overflow (full buffer), prevent underflow (empty buffer), and mutual exclusion on buffer access.

Semaphore mutex = 1; // mutual exclusion Semaphore empty = N; // counting: slots available (initially N) Semaphore full = 0; // counting: items in buffer (initially 0) Producer: Consumer: while(true) { while(true) { produce item; wait(full); wait(empty); wait(mutex); wait(mutex); item = remove from buffer; add item to buffer; signal(mutex); signal(mutex); signal(empty); signal(full); consume item; } }
The order matters! Always wait on mutex AFTER waiting on empty/full. If you wait(mutex) first, then wait(empty), you may hold mutex while sleeping → deadlock!
2. Readers-Writers Problem

Multiple readers can read simultaneously (safe). But a writer must have exclusive access. Two variants: (1) Reader priority — writers may starve; (2) Writer priority — readers may starve.

Semaphore rw_mutex = 1; // mutex for writers (and first/last reader) Semaphore mutex = 1; // mutex for read_count int read_count = 0; Writer: Reader: wait(rw_mutex); wait(mutex); [write] read_count++; signal(rw_mutex); if(read_count == 1) wait(rw_mutex); // first reader locks signal(mutex); [read] wait(mutex); read_count--; if(read_count == 0) signal(rw_mutex); // last reader unlocks signal(mutex);
3. Dining Philosophers Problem

5 philosophers sit at circular table, 5 forks between them. Each needs 2 forks (left and right) to eat. Problem: if all pick up left fork simultaneously → deadlock.

Semaphore fork[5] = {1,1,1,1,1}; Philosopher i: while(true) { think; wait(fork[i]); // pick up left fork wait(fork[(i+1)%5]); // pick up right fork eat; signal(fork[i]); // put down left fork signal(fork[(i+1)%5]); // put down right fork } /* PROBLEM: Deadlock if all pick up left fork! */

Solutions to Dining Philosophers Deadlock:

  • Allow at most 4 philosophers at the table simultaneously
  • Allow a philosopher to pick up forks only if BOTH are available (atomic)
  • Odd philosophers pick up left then right; even pick up right then left (asymmetric)

5.1 Deadlock Basics
What is Deadlock?
DeadlockA set of processes is in deadlock if every process in the set is waiting for an event that can only be caused by another process in the set. No process can proceed — permanent blocking.
📌 Classic Deadlock Scenario

P1 holds resource R1, waiting for R2. P2 holds R2, waiting for R1. Neither can proceed — deadlock!

4 Coffman Conditions (ALL must hold simultaneously)
  • 1. Mutual Exclusion — At least one resource must be held in non-shareable mode (only 1 process at a time)
  • 2. Hold and Wait — A process holds at least one resource and is waiting to acquire additional resources held by other processes
  • 3. No Preemption — Resources cannot be forcibly taken from a process; only voluntary release
  • 4. Circular Wait — A circular chain of processes exists: P0 waits for P1, P1 waits for P2, ..., Pn waits for P0
Break ANY ONE condition to prevent deadlock. The easiest to break: Circular Wait (impose ordering on resource acquisition). Hold and Wait can be broken by requiring all resources up front. Preemption: allow OS to forcibly take resources.
Resource Allocation Graph (RAG)

A directed graph with two types of nodes: Processes (circles) and Resources (rectangles with dots = instances).

  • Request edge: Process → Resource (process wants a resource)
  • Assignment edge: Resource → Process (resource is assigned to process)

Deadlock detection from RAG:

  • If graph has no cycle → NO deadlock
  • If graph has a cycle AND all resources have single instance → DEFINITELY deadlock
  • If graph has a cycle AND resources have multiple instances → POSSIBLY deadlock (need Banker's to confirm)
📌 RAG Example

R1 (1 instance): assigned to P1. R2 (2 instances): assigned to P1 and P2. P2 requesting R1. P3 requesting R2.

Cycle: P1 → R2 → P2 → R1 → P1? Let's trace: P1 holds R1, P2 holds R2, P2 requests R1 (held by P1), P1 requests R2 (one instance held by P2, one by P3 — wait, P3 holds R2? Re-check). If cycle exists with single-instance resources → deadlock.

Circular WaitImpose total ordering on all resources; processes must request in increasing orderMay be inconvenient; some orderings are not natural
Avoidance — Banker's Algorithm
Safe StateA state is safe if the system can allocate resources to each process in some order and still avoid deadlock. There exists a safe sequence — an ordering P1, P2, ..., Pn such that for each Pi, resources Pi still needs can be satisfied by currently available resources + resources held by Pj (j < i).

Data structures for Banker's Algorithm (n processes, m resources):

  • Available[m] — currently available instances of each resource
  • Max[n][m] — maximum demand of each process
  • Allocation[n][m] — currently allocated to each process
  • Need[n][m] = Max − Allocation — remaining need
Safety Algorithm: 1. Work = Available; Finish[i] = false for all i 2. Find i such that: Finish[i]=false AND Need[i] ≤ Work If no such i → go to step 4 3. Work = Work + Allocation[i]; Finish[i] = true; go to step 2 4. If Finish[i]=true for all i → SAFE STATE Otherwise → UNSAFE STATE
📌 Banker's Algorithm Example

Resources: A(10), B(5), C(7). 5 processes.

ProcessAllocation(A,B,C)Max(A,B,C)Need=Max-Alloc
P00,1,07,5,37,4,3
P12,0,03,2,21,2,2
P23,0,29,0,26,0,0
P32,1,12,2,20,1,1
P40,0,24,3,34,3,1

Total allocated: A=7, B=2, C=5. Total: A=10, B=5, C=7. Available = (3,3,2).

Safety check: Find process whose Need ≤ Available(3,3,2):

P1: Need(1,2,2) ≤ (3,3,2) ✓ → Allocate, Work becomes (3+2,3+0,2+0)=(5,3,2)

P3: Need(0,1,1) ≤ (5,3,2) ✓ → Work=(5+2,3+1,2+1)=(7,4,3)

P4: Need(4,3,1) ≤ (7,4,3) ✓ → Work=(7,4,5)

P0: Need(7,4,3) ≤ (7,4,5) ✓ → Work=(7,5,5)

P2: Need(6,0,0) ≤ (7,5,5) ✓ → Safe!

Safe sequence: P1 → P3 → P4 → P0 → P2

6.1 Address Binding & MMU
Logical vs Physical Address

MMU (Memory Management Unit)Hardware device that maps logical addresses to physical addresses at runtime. The mapping table is stored in registers (base/limit for simple schemes) or in page tables.
Address Binding — When does binding happen?
  • Compile time — If memory location is known at compile time, absolute code generated. Relocation requires recompilation. (e.g., .COM files in MS-DOS)
  • Load time — Compiler generates relocatable code. Binding done when OS loads program. If starting location changes, must reload.
  • Execution time — Binding delayed until runtime. Process can move during execution. Requires hardware support (MMU). Used in modern OS.
6.2 Contiguous Allocation
Fixed Partitioning

Memory divided into fixed-size partitions at system start. Each partition holds exactly one process.

  • Internal Fragmentation: Partition larger than process → wasted space inside partition. Example: 64KB partition, 40KB process → 24KB wasted inside.
  • Simple to implement but inflexible
Dynamic Partitioning

Partitions created dynamically of exact size needed. No internal fragmentation. But creates external fragmentation.

External Fragmentation: Total free memory is enough but not contiguous. Fix: Compaction (shuffle processes to create one large free block — expensive!).

Allocation Strategies
StrategyHowProsCons
First FitAllocate first hole big enough (from start)FastCreates small fragments at start of memory
Best FitAllocate smallest hole that is big enoughMinimizes wasted space in chosen holeCreates many tiny useless fragments; slow (scan all)
Worst FitAllocate largest holeLeftover fragment may be large enough to useBreaks up large holes; slow; generally worst performance
GATE: First Fit and Best Fit are generally better than Worst Fit. First Fit is fastest. Best Fit leaves smallest leftover. Neither is always optimal.
6.3 Paging
Paging Concept
PagingPhysical memory divided into fixed-size blocks called frames. Logical memory divided into same-size blocks called pages. OS maps pages to frames. Eliminates external fragmentation; may have small internal fragmentation.

Address translation:

Logical address = [Page Number (p) | Page Offset (d)] Physical address = [Frame Number (f) | Page Offset (d)] If page size = 2^n bytes: - Page number = top (log₂ total pages) bits of logical address - Offset = bottom n bits of logical address - Physical address = Frame_base + offset
📌 Paging Address Translation Example

Logical address space: 32 pages of 4 bytes each = 128 bytes total. Physical: 8 frames of 4 bytes = 32 bytes. Logical addr = 7 bits (5 for page, 2 for offset). Page table: P0→F5, P1→F6, P2→F1, P3→F2...

Logical address 11 (binary: 0001011): Page# = 00010 = 2, Offset = 11. Frame for P2 = F1 = 001. Physical addr = 001|11 = 00111 = 7.

TLB (Translation Lookaside Buffer)
TLBA hardware cache (associative memory) that stores recently used page-to-frame mappings. Makes address translation fast — if page# found in TLB (TLB hit), no need to access page table in memory.
Effective Access Time (EAT) with TLB: EAT = α × (TLB access time + memory access time) + (1-α) × (TLB access time + 2 × memory access time) Where α = TLB hit ratio (On TLB miss: 1 access for page table + 1 access for actual data = 2 memory accesses) Simplified (assuming TLB access negligible): EAT = α × m + (1-α) × 2m where m = memory access time
📌 EAT Calculation

Memory access time = 100ns. TLB access = 20ns. TLB hit rate = 80% (α = 0.8).

EAT = 0.8 × (20+100) + 0.2 × (20+100+100) = 0.8×120 + 0.2×220 = 96 + 44 = 140ns

Multilevel Paging

For large address spaces, page table itself becomes huge. Solution: Page the page table (hierarchical paging).

Two-level paging: Logical address = [p1 | p2 | d] p1 = outer page table index p2 = inner page table index d = offset within page For 32-bit address, 4KB pages: - 12 bits for offset (2^12 = 4096 = 4KB) - 20 bits for page number → split as 10|10 - 3 memory accesses: outer PT + inner PT + actual data
Key formula: For k-level page table, TLB miss costs k+1 memory accesses. With 2-level paging: TLB miss = 3 accesses (outer PT + inner PT + data). With TLB hit = 1 access.

7.1 Demand Paging
Demand PagingPages are loaded into memory only when needed (on demand). A page not in memory has its valid bit = 0 in page table. Accessing such a page causes a page fault.

Page Fault Handling Steps:

  1. Process accesses memory → MMU checks page table → valid bit = 0 → PAGE FAULT trap to OS
  2. OS checks internal table: valid reference? If invalid → terminate process
  3. OS finds free frame in physical memory
  4. OS issues disk I/O to read the page into the frame
  5. OS updates page table: frame number + valid bit = 1
  6. OS restarts the interrupted instruction
  7. Process continues as if no fault occurred
Effective Access Time with Page Faults: EAT = (1 - p) × ma + p × page_fault_time Where p = page fault rate (0 ≤ p ≤ 1) ma = memory access time (typically 100-200 ns) page_fault_time = page fault service time (typically 8-20 ms = millions of ns!)
📌 EAT Calculation

ma = 200ns. Page fault time = 8ms = 8,000,000ns. Desired EAT ≤ 220ns (10% slowdown). Find max p.

220 = (1-p)×200 + p×8,000,000

220 = 200 - 200p + 8,000,000p

20 = 7,999,800p → p = 20/7,999,800 ≈ 1/400,000

Only 1 page fault per 400,000 accesses is acceptable!

Goal: Minimize the number of page faults.

7.3 Thrashing
ThrashingA process is thrashing if it spends more time paging (in and out) than executing. Occurs when a process doesn't have enough frames for its working set. CPU utilization drops drastically while disk I/O is very high.

Why thrashing happens: OS sees CPU idle → increases degree of multiprogramming → adds more processes → each process has fewer frames → more page faults → less CPU time for actual execution → CPU even more idle → vicious cycle.

Working Set Model: The working set of a process is the set of pages it actively uses in a recent window of time Δ (working set window). If total working sets of all processes exceed available frames → thrashing will occur.

Working Set Window (Δ) = most recent Δ page references WSS_i(t) = number of distinct pages in working set of process i at time t D = Σ WSS_i(t) (total working set demand) If D > m (total frames) → suspend/swap out a process to reduce D

8.3 File Allocation Methods
1. Contiguous Allocation

Each file occupies a set of contiguous disk blocks. Directory entry stores start block and length.

  • Pros: Excellent sequential access (one disk seek), direct access possible (block i = start + i), simple
  • Cons: External fragmentation, file size must be known at creation, difficult to grow file
2. Linked Allocation

Each file is a linked list of disk blocks. Directory has pointer to first block; each block has pointer to next. Last block has null pointer.

  • Pros: No external fragmentation, file can grow easily, no need to declare size
  • Cons: No efficient direct access (must traverse from start), pointer overhead per block, reliability (broken pointer loses rest of file)
FAT (File Allocation Table) is a variation of linked allocation where all pointers are stored in a table in memory rather than in the blocks themselves.
3. Indexed Allocation

Each file has an index block containing pointers to all its data blocks. Directory points to index block.

  • Pros: Direct access (jump to i-th block directly), no external fragmentation
  • Cons: Index block overhead; large files need multiple index blocks (linked/multilevel/combined scheme)
📌 Combined Inode Structure (UNIX)

UNIX inode has: 12 direct block pointers + 1 single indirect + 1 double indirect + 1 triple indirect.

With 4KB blocks and 4-byte pointers → 1024 pointers per block:

  • Direct: 12 × 4KB = 48KB
  • Single indirect: 1024 × 4KB = 4MB
  • Double indirect: 1024 × 1024 × 4KB = 4GB
  • Triple indirect: 1024³ × 4KB = 4TB
GATE questions often ask: "How many disk accesses needed to read block number N of a file?" Account for inode access + possible indirect block accesses + final data block access.

9.2 Disk Scheduling Algorithms
Disk Access Time = Seek Time + Rotational Latency + Transfer Time. Disk scheduling algorithms minimize seek time by reordering service of disk I/O requests.
1. FCFS (First Come First Served)

Service requests in order they arrive. Fair but not optimal.

📌 Example — Initial head position: 53, Queue: 98,183,37,122,14,124,65,67

FCFS order: 53→98→183→37→122→14→124→65→67

Total head movement = |98-53|+|183-98|+|37-183|+|122-37|+|14-122|+|124-14|+|65-124|+|67-65|

= 45+85+146+85+108+110+59+2 = 640 cylinders

2. SSTF (Shortest Seek Time First)

Service the request closest to current head position. Reduces seek time significantly but may cause starvation for far-away requests.

📌 SSTF — Head=53, Queue: 98,183,37,122,14,124,65,67

From 53: closest = 65. 65→67→37→14→98→122→124→183

Movement = 12+2+30+23+84+24+2+59 = 236 cylinders

3. SCAN (Elevator Algorithm)

Head moves in one direction servicing all requests, reaches end of disk (or last request), reverses direction. Like an elevator.

📌 SCAN — Head=53 moving toward 0, Queue: 98,183,37,122,14,124,65,67

Move toward 0: service 37,14,0(end), then reverse: 65,67,98,122,124,183

Movement = |37-53|+|14-37|+|0-14|+|65-0|+|67-65|+... = 16+23+14+65+2+31+24+2+59 = 236

4. C-SCAN (Circular SCAN)

Like SCAN but only services requests in ONE direction. After reaching end, jumps back to start (no service on return). More uniform waiting time than SCAN.

5. LOOK & C-LOOK

Variants of SCAN/C-SCAN that only go as far as the last request in that direction (not to the physical end of disk). More practical than SCAN/C-SCAN.
